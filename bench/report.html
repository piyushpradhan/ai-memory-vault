Design decisions

  • MiniLM over BGE — BGE‑small‑en‑v1.5 scored 50% on this vault (vs MiniLM's 70%). BGE is tuned for passage retrieval on MS MARCO‑style datasets; this vault contains short bullet‑point markdown notes. Domain mismatch negates the MTEB advantage.
  • No cross‑encoder for retrieval — ms‑marco‑MiniLM‑L‑6‑v2 dropped accuracy to 45% by favoring long expository text over focused bullet‑point notes. Cross‑encoder is retained for check_overlaps near‑duplicate detection where it matches correctly (99.4% on tool‑list test).
  • Vector‑first, BM25 gap‑fill — pure RRF‑style weighted fusion dilutes vector rankings (pushed 5 previously‑correct hits out of the top‑3). Gap‑fill preserves the vector order and only adds BM25‑exclusive results at the end for vocabulary‑mismatch cases.
  • Per‑note dedup — eliminates the common case where multi‑section notes occupy 2 of 3 result slots. Keeps the best‑scoring chunk per source file.

Comparison with SOTA memory products

Mem0 — 60.1k stars, Apache 2.0, YC S24

Mem0 is the dominant memory‑layer SDK for AI agents. Its April 2026 algorithm posts 91.6% LoCoMo and 94.8% LongMemEval. It uses a server‑side LLM (defaults to OpenAI) for ADD‑only extraction — one LLM call extracts facts from conversations, never UPDATE/DELETE. Facts accumulate as discrete memory entries in a proprietary graph structure.

Strengths

  • Entity linking — entities are extracted, embedded, and cross‑linked across memories for retrieval boosting.
  • Multi‑signal retrieval — semantic + BM25 keyword + entity matching fused in parallel.
  • Temporal reasoning — time‑aware retrieval that ranks the right dated instance for current‑state vs past‑event queries.
  • LLM‑driven extraction — the server‑side extractor produces high‑quality, normalized facts.
  • Full platform — cloud dashboard, auth, API keys, self‑hosted server, npm/pip SDKs.

Trade‑offs vs ai‑memory

  • Server‑side LLM required — mem0 cannot run without an API key to OpenAI or compatible provider. ai‑memory runs fully offline with local embeddings only.
  • Proprietary graph storage — memories live in mem0's internal structure, not human‑editable. ai‑memory stores plain markdown files in an Obsidian vault.
  • Extraction latency — each add() triggers an LLM call (~0.8–1s). ai‑memory writes a markdown file directly (~ms).
  • Not a notebook — mem0 has no UI; ai‑memory doubles as an Obsidian vault (human‑browsable, git‑trackable).
  • LoCoMo gap — mem0's 91.6% vs ai‑memory's 50%. The 41.6pp delta comes entirely from the LLM extractor, not the retrieval engine.

Supermemory — 28.2k stars, MIT

Supermemory is a full‑stack memory engine + web app claiming #1 on LongMemEval (81.6%), LoCoMo, and ConvoMem. It goes beyond retrieval into knowledge management: connectors (Google Drive, Gmail, Notion, GitHub), multi‑modal extractors (PDFs, images, video, code AST), and auto‑maintained user profiles with static + dynamic facts.

Strengths

  • User profiles — one‑call profile API returns stable facts + recent context in ~50ms. No query needed.
  • Connectors — auto‑sync external data with real‑time webhooks. Not just memories — a full knowledge base.
  • Hybrid search — RAG + Memory in one query. Knowledge docs and personalized context together.
  • Multi‑modal — processes PDFs, images (OCR), videos (transcription), code (AST‑aware chunking).
  • Local binary — one‑command install, runs fully offline with Ollama. Bring any model.
  • MemoryBench — open‑source benchmarking framework for comparing memory providers head‑to‑head.

Trade‑offs vs ai‑memory

  • Complexity — full‑stack app with Postgres, Remix, Next.js, Cloudflare Workers. ai‑memory is ~500 lines of Python in a single package.
  • Heavy stack — needs Docker or a Node runtime for the self‑hosted binary. ai‑memory runs as a single aimemory‑server process.
  • Data portability — memories live in Supermemory's ontology/tables. ai‑memory is plain markdown — copy the vault directory, done.
  • Token footprint — Supermemory serves user profiles + search results — context can be large. ai‑memory enforces a 1,500‑char budget with sentence‑boundary truncation.
  • Benchmark profile — benchmarks on standard datasets, same mem0‑style extraction pipeline. No public custom‑vault eval results for personal markdown notes.

OpenClaw — 382k stars, MIT

OpenClaw is a personal AI assistant platform, not primarily a memory product. Its memory capabilities come from the agent framework layer — BM25 + vector hybrid search, session‑scoped context windows, and workspace‑based skill injection (AGENTS.md, SOUL.md). It is the closest analog to ai‑memory's use case: a local‑first, single‑user assistant with persistent context.

Strengths

  • Local‑first Gateway — single control plane on your own devices. Same philosophy as ai‑memory.
  • Multi‑channel — WhatsApp, Telegram, Slack, Discord, Signal, iMessage, and 15+ more. Not just an API.
  • Voice + Canvas — macOS/iOS/Android apps with wake words, talk mode, and a live agent‑driven canvas.
  • Plugin ecosystem — ClawHub skill registry, Supermemory integration plugin available.
  • Sandboxing — per‑session Docker/SSH sandboxes for channel safety. Production‑grade security model.

Trade‑offs vs ai‑memory

  • Product, not a tool — OpenClaw is a full personal assistant with 64k+ commits over 2+ years. ai‑memory is a focused memory layer you plug into any agent.
  • No standalone memory API — OpenClaw's memory is session‑scoped within its gateway. You can't use it as a drop‑in memory backend for Claude Code or Cursor.
  • Memory is not the product — retrieval, extraction, consolidation, and forgetting are agent behaviors, not a dedicated memory pipeline. No published memory benchmarks.
  • Uses Supermemory for memory — OpenClaw has a Supermemory plugin. Its standalone memory capabilities rely on the Gateway context window (AGENTS.md injection), not a semantic index.
  • ai‑memory is an MCP drop‑in — 3 lines of config, works with Claude Code/Desktop, OpenCode, Grok, any MCP client. OpenClaw needs its own Gateway running.

Where ai‑memory fits

ai‑memory is the MCP‑native, markdown‑backed, offline‑first memory layer for personal AI assistants. It doesn't compete on benchmark scores — it competes on being the smallest, simplest, most portable memory backend you can drop into any MCP‑compatible agent in 3 lines of config. The other products are platforms; ai‑memory is a tool.
  • vs mem0: If you need benchmark‑topping accuracy and can pay for LLM extraction, use mem0. If you need zero‑latency local memory with human‑editable files, use ai‑memory.
  • vs Supermemory: If you need a full knowledge management platform with connectors and user profiles, use Supermemory. If you need a focused memory layer for an existing agent, use ai‑memory.
  • vs OpenClaw: If you want a complete personal assistant with multi‑channel support, use OpenClaw. If you already have an agent and just need its memory layer, use ai‑memory.

The 41.6pp LoCoMo gap vs mem0 is entirely due to the LLM extractor — mem0's server‑side LLM normalizes and deduplicates facts before storage. ai‑memory intentionally pushes extraction to the calling agent (the agent is the LLM) to stay offline. Multi‑hop failures in ai‑memory's LOCOMO run were answerer‑side (DeepSeek returning raw JSON), not retrieval‑side.
